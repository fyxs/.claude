## 策略性上下文压缩

> **灵感来源**: ECC strategic-compact 技能

### 核心理念

**关键原则：避免在上下文窗口的最后 20% 进行复杂操作。**

**策略性压缩 vs 自动压缩**：

- ❌ **自动压缩问题**：在任意点触发，可能在任务中途丢失重要上下文
- ✅ **策略性压缩优势**：在逻辑边界压缩，保留关键信息，清理无用上下文

### 何时建议 /compact

| 阶段切换 | 是否压缩 | 原因 |
|----------|---------|------|
| 探索研究 → 制定计划 | ✅ 是 | 研究上下文体积大，计划是精炼输出 |
| 制定计划 → 编码实现 | ✅ 是 | 计划已记录在 TodoWrite 或文件中 |
| 编码实现 → 运行测试 | ⚠️ 视情况 | 如果测试引用最近代码则保留 |
| 调试修复 → 开发新功能 | ✅ 是 | 调试信息会污染新任务的上下文 |
| 实现进行中 | ❌ 否 | 丢失变量名、文件路径、部分状态代价高 |
| 失败方案后 | ✅ 是 | 清理无效推理再尝试新方案 |
| 完成里程碑后 | ✅ 是 | 为下一阶段提供干净的起点 |
| 上下文压力（响应变慢） | ✅ 是 | 接近上下文限制时主动压缩 |

### 压缩触发信号

**何时考虑压缩：**

1. **工具调用次数** - 会话中工具调用超过 50 次
2. **上下文窗口** - 使用率超过 60%
3. **响应质量** - 响应变慢或不连贯
4. **任务边界** - 完成一个主要任务，开始新任务
5. **失败尝试** - 某个方案失败，需要尝试新方法

**压缩决策：**
```
检测到触发信号？
    ↓ 是
在任务边界？
    ↓ 是
关键信息已持久化？
    ↓ 是
    → 执行 /compact
    ↓ 否
    → 先持久化，再压缩
```

### 压缩最佳实践

**1. 使用带摘要的压缩**
```bash
# 不要只用 /compact
/compact

# 使用带摘要的压缩
/compact 专注于实现认证中间件，已完成用户模型和数据库迁移
```

**2. 压缩前的检查清单**
- [ ] TodoWrite 已更新当前进度
- [ ] 关键决策已写入文件或注释
- [ ] 下一步计划已明确记录
- [ ] 重要的变量名、文件路径已记录

**3. 压缩时机判断**
```markdown
✅ 好的压缩时机：
- 完成探索，开始实施
- 完成规划，开始编码
- 完成调试，开始新功能
- 失败方案后，尝试新方法

❌ 差的压缩时机：
- 正在编写复杂函数
- 正在调试多个相关文件
- 刚开始理解代码库
- 需要引用最近的讨论
```

## Token 使用优化习惯

### 输出控制

**核心原则：** 减少不必要的输出，避免重复

1. **代码输出优先简洁**
   ```markdown
   ❌ 差的做法：
   - 重复显示整个文件
   - 输出已知的代码
   - 冗长的解释

   ✅ 好的做法：
   - 只显示变更部分
   - 引用文件路径和行号
   - 简洁的说明
   ```

2. **使用 Edit 而非 Write 重写整个文件**
   ```markdown
   场景：修改文件中的一行

   ❌ 使用 Write：
   - 读取整个文件（1000 tokens）
   - 写入整个文件（1000 tokens）
   - 总计：2000 tokens

   ✅ 使用 Edit：
   - 读取整个文件（1000 tokens）
   - 只写入变更部分（50 tokens）
   - 总计：1050 tokens
   - 节省：47.5%
   ```

3. **长文件分段读取**
   ```javascript
   // 使用 offset 和 limit 参数
   Read(file_path, offset=0, limit=100)    // 读取前 100 行
   Read(file_path, offset=100, limit=100)  // 读取接下来 100 行
   ```

---

### 工具使用

**核心原则：** 减少往返次数，提高效率

1. **合并无依赖的工具调用**
   ```markdown
   ❌ 顺序调用（3 次往返）：
   1. Read file1.js
   2. Read file2.js
   3. Read file3.js

   ✅ 并行调用（1 次往返）：
   1. Read file1.js + Read file2.js + Read file3.js
   ```

2. **优先使用 Grep/Glob 定位**
   ```markdown
   场景：查找包含 "authenticate" 的文件

   ❌ 低效方式：
   1. Glob "**/*.js" → 100 个文件
   2. Read 每个文件检查内容
   3. 总计：100+ 次工具调用

   ✅ 高效方式：
   1. Grep "authenticate" --type js
   2. 总计：1 次工具调用
   ```

3. **用 Task(model:haiku) 处理简单子任务**
   ```markdown
   场景：格式化 10 个文件

   ❌ 使用主 Agent (Sonnet)：
   - 成本：$3.00 / 1M tokens

   ✅ 使用子 Agent (Haiku)：
   - 成本：$0.80 / 1M tokens
   - 节省：73%
   ```

---

## Prompt Caching 机制

### 缓存工作原理

**核心概念：**

1. **缓存阈值：≥1024 tokens**
   - 小于 1024 tokens 的内容不会被缓存
   - 大于等于 1024 tokens 的内容会被缓存

2. **缓存有效期**
   - 短期缓存：5 分钟
   - 长期缓存：1 小时（取决于使用频率）

3. **缓存命中节省**
   - 缓存命中可节省 90% 输入 token 成本
   - 例如：10K tokens 缓存命中，成本从 $0.03 降至 $0.003

---

### 缓存边界设计

**最佳实践：**

```
[System Prompt - 缓存] ← 稳定内容
[Rules Files - 缓存]   ← 稳定内容
[Project Context - 缓存] ← 相对稳定
─────────────────────── 缓存边界
[User Message - 不缓存] ← 动态内容
[Dynamic Context - 不缓存] ← 动态内容
```

**优化原则：**
1. 将稳定内容放在前面
2. 将动态内容放在后面
3. 避免在缓存内容中插入时间戳或随机值
4. 保持缓存内容的稳定性

---

### 缓存失效触发因素

**会导致缓存失效的操作：**

1. **规则文件修改**
   - 修改 .claude/rules/ 中的任何文件
   - 添加或删除规则文件

2. **超过缓存时间**
   - 5 分钟未使用（短期缓存）
   - 1 小时未使用（长期缓存）

3. **上下文前缀变化**
   - 系统提示变化
   - 规则加载顺序变化

**避免缓存失效：**
- 保持规则文件稳定
- 频繁使用以保持缓存热度
- 避免不必要的规则文件修改

---

### 文件大小优化

**规则文件大小建议：**

| 文件大小 | 缓存效果 | 建议 |
|---------|---------|------|
| < 1024 tokens | ❌ 不缓存 | 合并到其他文件 |
| 1024-1500 tokens | ⚠️ 边缘 | 扩充到 1500+ |
| 1500-3000 tokens | ✅ 理想 | 保持此范围 |
| > 3000 tokens | ⚠️ 过大 | 考虑拆分 |

**优化策略：**
```markdown
1. 合并小文件（< 1024 tokens）
2. 扩充边缘文件（1024-1500 tokens）
3. 保持理想大小（1500-3000 tokens）
4. 拆分过大文件（> 3000 tokens）
```

## 最佳实践总结

### 上下文管理
1. **主动压缩** - 在 60% 时压缩，不要等到 90%
2. **持久化关键信息** - 使用 TodoWrite 和文件
3. **清理无效推理** - 失败方案后立即压缩

### 工具使用
1. **并行调用** - 合并无依赖的工具调用
2. **精准定位** - 使用 Grep/Glob 而非全文读取
3. **增量编辑** - 使用 Edit 而非 Write

### 模型选择
1. **任务分类** - 根据复杂度选择模型
2. **成本意识** - 优先使用 Haiku 处理简单任务
3. **质量保证** - 复杂任务使用 Sonnet/Opus

### 缓存优化
1. **文件大小** - 保持规则文件在 1500-3000 tokens
2. **稳定性** - 避免频繁修改规则文件
3. **结构化** - 将稳定内容放在前面

### 监控度量
1. **跟踪指标** - 记录 token 使用和成本
2. **定期审查** - 每周/每月检查优化机会
3. **持续改进** - 根据数据调整策略

**记住：** Token 优化是一个持续的过程。通过监控、分析和调整，可以显著降低成本并提高效率。
