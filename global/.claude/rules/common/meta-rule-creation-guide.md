# 元规则：规则创建指南

> 规则 ID: META-001
> 版本: 1.0.0
> 创建日期: 2026-03-09
> 状态: 发布
> 强制级别: CRITICAL

## 目的

本规则指导 Claude 如何创建、修改和维护规则文件，确保规则质量和一致性。

**CRITICAL**：每次创建或修改规则时，必须遵循本指南。

---

## 规则创建流程

### 阶段 1：需求分析

**触发条件**：
- 发现规则执行错误
- 用户提出新的行为要求
- 现有规则不够清晰

**必须回答的问题**：
- [ ] 这个规则要解决什么问题？
- [ ] 为什么需要这个规则？
- [ ] 是否与现有规则冲突？
- [ ] 是否有更简单的解决方案？
- [ ] 规则的适用范围是什么？

**输出**：明确的规则需求描述

### 阶段 2：规则设计

**设计原则**：
1. **SMART 原则**
   - Specific（具体的）
   - Measurable（可衡量的）
   - Achievable（可实现的）
   - Relevant（相关的）
   - Time-bound（有时限的）

2. **主题聚焦（平衡单一职责与缓存优化）**

   **核心原则**：一个规则文件负责一个**主题**，而非一个**规则**。

   **文件大小指导**：
   - **理想范围**：1500-3000 tokens
   - **最小阈值**：1024 tokens（缓存要求）
   - **最大限制**：5000 tokens（避免过大）

   **合并策略**：

   ✅ **应该合并**（相关规则，同一主题）：
   - 多个工具使用规则 → `tools-usage.md`
   - TDD + 代码审查 + Git 工作流 → `development-workflow.md`
   - 代码标准 + 性能 + 安全 → `standards-performance-security.md`

   ❌ **不应该合并**（不相关规则，不同主题）：
   - NotebookLM 使用 + Edge 浏览器自动化（不同工具）
   - 开发工作流 + Token 优化（不同领域）

   **判断标准**：
   - 规则之间是否有逻辑关联？
   - 是否属于同一个工作流程？
   - 用户是否会同时需要这些规则？

   **缓存优化考虑**：
   - 小于 1024 tokens 的规则应该合并到相关主题文件
   - 稳定的规则（很少修改）优先合并
   - 经常修改的规则保持独立

3. **显式优于隐式**
   - 明确说明，不依赖推断
   - 提供具体示例

4. **最小惊讶**
   - 规则行为符合直觉
   - 避免意外的副作用

**输出**：规则设计草稿

## 规则命名规范

**格式**：`{层级}-{主题}-{具体内容}.md`

**层级前缀**：
- `global-` - 全局层规则
- `domain-` - 领域层规则
- `app-` - 应用层规则
- `project-` - 项目层规则
- `meta-` - 元规则（关于规则的规则）

**主题分类**：
- `workflow-` - 工作流规则
- `tool-` - 工具使用规则
- `standard-` - 标准规范规则
- `security-` - 安全规则
- `performance-` - 性能规则

**示例**：
- `domain-workflow-tdd.md` - 领域层的 TDD 工作流规则
- `meta-rule-creation-guide.md` - 元规则：规则创建指南

**例外（特殊文件）**：
- `architecture.md` - 架构说明文件，作为入口文档不加前缀
- `INDEX.md` - 规则索引文件，不加前缀

**触发条件**：

7. 更新相关文档

**版本号规则**：
- 主版本号：不兼容的重大变更
- 次版本号：向后兼容的功能性新增
- 修订号：向后兼容的问题修正

## 规则冲突处理

**优先级规则**：
1. **前缀规则** > 关键词规则
2. **下层规则** > 上层规则（项目 > 应用 > 领域 > 全局）
3. **CRITICAL** > MANDATORY > RECOMMENDED > OPTIONAL
4. **显式规则** > 隐式规则
5. **具体规则** > 通用规则

**冲突预防**：
- 在规则中明确说明优先级
- 说明与其他规则的关系
- 使用"除非"条款处理例外

**示例**：
```markdown
**优先级**：最高（覆盖所有其他工具选择规则）

**例外**：无

**与其他规则的关系**：
- 覆盖：NotebookLM 自动触发规则
- 不影响：内容分类规则
```

## 常见错误与避免

### 错误 1：规则描述模糊

**错误示例**：
```markdown
使用 Obsidian 记录信息
```

**正确示例**：
```markdown
当用户消息以 "obsidian" 开头时，使用 Obsidian 记录内容。
必须结合对话上下文判断主题。
```

### 错误 2：缺少示例

**错误示例**：
```markdown
前缀规则优先
```

**正确示例**：
```markdown
前缀规则优先

示例：
- "obsidian 深入探讨" → 使用 Obsidian（前缀规则）
- "深入探讨" → 根据关键词判断（关键词规则）
```

### 错误 3：没有测试用例

**错误**：规则发布后没有测试用例

**正确**：每个规则至少 3 个测试用例

### 错误 4：规则过于宽泛

**错误示例**：
```markdown
# 工具使用规则

包含所有工具的使用规则...
```

**正确示例**：
```markdown
# NotebookLM 使用规则

只定义 NotebookLM 的使用规则
```
