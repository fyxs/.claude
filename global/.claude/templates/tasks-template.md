# 任务列表：[功能名称]

**输入**：来自 `/specs/[###_分支name_任务或功能name]/` 的设计文档
**前提**：plan.md（必须）、spec.md（用户故事必须）、research.md、data-model.md、contracts/

**测试**：以下示例包含测试任务。测试为可选项——仅在功能规格中明确要求时才包含。

**组织方式**：任务按用户故事分组，以便每个故事可独立实现和测试。

## 格式：`[ID] [P?] [故事] 描述`

- **[P]**：可并行执行（不同文件，无依赖关系）
- **[故事]**：任务所属用户故事（如 US1、US2、US3）
- 描述中包含具体文件路径

## 路径约定

- **单项目**：`src/`、`tests/` 位于仓库根目录
- **Web 应用**：`backend/src/`、`frontend/src/`
- **移动端**：`api/src/`、`ios/src/` 或 `android/src/`
- 以下路径以单项目为例，请根据 plan.md 结构调整

## 依赖关系与执行顺序

### 阶段依赖

- **初始化（阶段 1）**：无依赖，可立即开始
- **基础建设（阶段 2）**：依赖初始化完成——阻塞所有用户故事
- **用户故事（阶段 3+）**：均依赖基础建设完成
  - 各用户故事可并行推进（如有足够人手）
  - 或按优先级顺序推进（P1 → P2 → P3）
- **收尾（最终阶段）**：依赖所有目标用户故事完成

### 用户故事依赖

- **用户故事 1（P1）**：基础建设完成后可开始，不依赖其他故事
- **用户故事 2（P2）**：基础建设完成后可开始，可与 US1 集成但应独立可测
- **用户故事 3（P3）**：基础建设完成后可开始，可与 US1/US2 集成但应独立可测

### 每个用户故事内部顺序

- 测试（如包含）必须先编写并确认失败，再开始实现
- 模型先于服务
- 服务先于接口
- 核心实现先于集成
- 本故事完成后再推进下一优先级

### 并行机会

- 所有标记 [P] 的初始化任务可并行执行
- 所有标记 [P] 的基础建设任务可在阶段 2 内并行执行
- 基础建设完成后，所有用户故事可并行开始（视团队容量）
- 同一用户故事内标记 [P] 的测试任务可并行执行
- 同一故事内标记 [P] 的模型任务可并行执行
- 不同用户故事可由不同团队成员并行推进

## 实施策略

### MVP 优先（仅用户故事 1）

1. 完成阶段 1：初始化
2. 完成阶段 2：基础建设（关键——阻塞所有故事）
3. 完成阶段 3：用户故事 1
4. **停止并验证**：独立测试用户故事 1
5. 如已就绪，部署/演示

### 增量交付

1. 完成初始化 + 基础建设 → 基础就绪
2. 添加用户故事 1 → 独立测试 → 部署/演示（MVP！）
3. 添加用户故事 2 → 独立测试 → 部署/演示
4. 添加用户故事 3 → 独立测试 → 部署/演示
5. 每个故事在不破坏前序故事的前提下独立增加价值

### 团队并行策略

多人协作时：

1. 团队共同完成初始化 + 基础建设
2. 基础建设完成后：
   - 开发者 A：用户故事 1
   - 开发者 B：用户故事 2
   - 开发者 C：用户故事 3
3. 各故事独立完成并集成

## 备注

- [P] 任务 = 不同文件，无依赖关系
- [故事] 标签将任务映射到具体用户故事，便于追溯
- 每个用户故事应可独立完成和测试
- 实现前确认测试失败
- 每个任务或逻辑组完成后提交
- 在任意检查点停下来独立验证故事
- 避免：任务描述模糊、同文件冲突、破坏故事独立性的跨故事依赖
