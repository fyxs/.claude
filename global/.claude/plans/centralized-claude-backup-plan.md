# Claude 配置集中备份方案计划

> 计划主题：四层架构 Claude 配置的集中仓库备份与恢复
> 适用范围：全局层 / 领域层 / 应用层 / 项目层
> 输出位置：`C:\Users\admin\.claude\plans\centralized-claude-backup-plan.md`
> 状态：草案
> 原则：先规划，不直接实施

---

## 1. 目标

建立一套适用于四层架构的 Claude 配置集中备份方案，满足以下目标：

1. 防止 `\.claude` 目录因误删、覆盖、磁盘故障而无法恢复
2. 用一个集中仓库统一收纳多个 `.claude` 目录的备份副本
3. 保持备份结构与四层架构一致，便于理解、审查、恢复与迁移
4. 采用单向同步，避免双向自动同步带来的覆盖风险
5. 同时具备：
   - Git 版本历史能力
   - 本地快照归档能力
   - 后续扩展到云端或私有远程仓库的能力

---

## 2. 设计原则

### 2.1 架构对齐

集中备份仓库必须按四层结构组织，而不是按磁盘散乱路径组织。

### 2.2 单向同步

默认只允许：

`源 .claude 目录 -> 集中备份仓库`

不默认允许：

`集中备份仓库 -> 源目录`

恢复操作必须作为单独流程执行。

### 2.3 Manifest 驱动

所有源路径、目标路径、层级归属都通过清单文件统一管理，避免把路径硬编码分散在多个脚本中。

### 2.4 Git + 快照双保险

- Git：负责文本级版本历史、变更追踪、回滚
- Zip 快照：负责灾难恢复和历史归档

### 2.5 安全最小化

集中仓库只保留应备份内容；临时文件、缓存、日志、敏感凭证文件默认排除。

---

## 3. 推荐整体方案

### 3.1 集中备份仓库

建议建立一个专用目录作为集中备份仓库，例如：

`D:\Backup\claude-config-backup`

该目录初始化为 Git 仓库，并可选连接到私有远程仓库。

### 3.2 仓库内部结构

建议结构：

```text
claude-config-backup/
├─ README.md
├─ manifest/
│  ├─ sources.json
│  └─ restore-notes.md
├─ global/
│  └─ .claude/
├─ domains/
│  └─ <domain-name>/.claude/
├─ apps/
│  └─ <app-name>/.claude/
├─ projects/
│  └─ <project-name>/.claude/
└─ .gitignore
```

### 3.3 路径映射清单

通过 `manifest/sources.json` 定义：

- 配置层级
- 源目录路径
- 备份仓库内目标路径
- 是否启用同步
- 可选说明信息

示例结构：

```json
{
  "global": [
    {
      "name": "global",
      "source": "C:\\Users\\admin\\.claude",
      "target": "global\\.claude",
      "enabled": true
    }
  ],
  "domains": [],
  "apps": [],
  "projects": []
}
```

## 4. 分阶段实施计划

## 阶段 A：确定备份边界

### 目标
明确哪些 `.claude` 目录需要纳入集中备份，以及各自属于哪一层。

### 任务
1. 盘点所有现有 `.claude` 目录
2. 为每个目录标注所属层级：
   - global
   - domains
   - apps
   - projects
3. 为每个目录定义逻辑名称
4. 确认是否存在不应进入备份的目录

### 产出
- 一份完整的 `.claude` 目录清单
- 各目录的层级归属和命名规则

---

## 阶段 B：设计集中仓库结构

### 目标
建立集中仓库目录结构，使之与四层架构一致。

### 任务
1. 创建集中备份根目录
2. 规划 `global/`、`domains/`、`apps/`、`projects/` 子目录
3. 规划 `manifest/` 目录
4. 确定命名规范：
   - domain 名称
   - app 名称
   - project 名称
5. 编写 `README.md` 说明仓库职责和结构

### 产出
- 集中仓库骨架目录
- 命名规范
- 仓库说明文档

---

## 阶段 C：设计 Manifest 清单

### 目标
通过清单文件实现备份配置集中管理。

### 任务
1. 设计 `manifest/sources.json` 结构
2. 为每个 `.claude` 目录添加映射
3. 定义字段：
   - `name`
   - `source`
   - `target`
   - `enabled`
   - `notes`（可选）
4. 约定新增目录时的维护流程

### 产出
- `manifest/sources.json`
- 清单字段说明

---

## 阶段 D：实现单向同步脚本

### 目标
实现从多个源 `.claude` 目录同步到集中仓库的自动化脚本。

### 任务
1. 选择脚本方案：PowerShell 优先
2. 脚本读取 `manifest/sources.json`
3. 对每个启用项执行同步：
   - 若源目录不存在，则记录警告
   - 若存在，则同步到对应目标目录
4. 同步时排除不需要内容，例如：
   - `temp/`
   - 日志文件
   - 缓存文件
5. 输出同步结果汇总

### 技术建议
- 文件复制可优先用 `robocopy`
- 清单解析使用 PowerShell JSON 支持
- 脚本需要可重复执行（幂等）

### 产出
- `sync-claude-configs.ps1`
- 同步结果日志格式

---

## 阶段 E：初始化 Git 版本管理

### 目标
让集中备份仓库具备版本历史与回滚能力。

### 任务
1. 在集中仓库执行 `git init`
2. 编写 `.gitignore`
3. 首次同步后提交初始版本
4. 可选：连接私有远程仓库
5. 约定提交频率和提交消息规范

### Git 建议
提交信息可采用：

```text
chore: sync claude configs
```

如需要更精细，也可按层级写：

```text
chore: sync global and project claude configs
```

### 产出
- Git 仓库初始化完成
- `.gitignore`
- 首次提交完成

---

## 阶段 F：建立快照归档机制

### 目标
在 Git 之外，再保留定期压缩快照，用于灾难恢复。

### 任务
1. 规划快照目录，例如：
   `D:\Backup\claude-config-archives`
2. 编写压缩脚本，对集中仓库或源配置进行归档
3. 制定保留策略，例如：
   - 最近 7 天：每日一份
   - 最近 4 周：每周一份
   - 最近 6~12 个月：每月一份
4. 可选：同步快照到云盘或私有存储

### 产出
- 快照脚本
- 快照目录结构
- 快照保留策略说明

---

## 阶段 G：建立恢复流程

### 目标
确保备份不仅存在，而且可实际恢复。

### 任务
1. 编写 `manifest/restore-notes.md`
2. 规定恢复场景：
   - 恢复单个项目层 `.claude`
   - 恢复某一层全部配置
   - 恢复整个集中仓库
3. 说明恢复前的检查步骤
4. 明确恢复为人工确认流程，不自动覆盖
5. 设计一次恢复演练

### 产出
- 恢复说明文档
- 恢复演练清单

## 6. 风险与对策

### 风险 1：把不该备份的敏感文件提交进 Git
**对策：**
- 先人工审查各 `.claude` 目录
- 补充 `.gitignore`
- 私有仓库也不默认信任

### 风险 2：镜像同步导致误删除传播
**对策：**
- Git 保留历史
- 额外保留 Zip 快照
- 不只依赖镜像目录

### 风险 3：集中仓库结构与四层逻辑脱节
**对策：**
- 按层组织目录
- 用 manifest 明确映射
- 统一命名规范

### 风险 4：恢复时误覆盖现有目录
**对策：**
- 恢复作为单独流程
- 恢复前人工确认
- 优先恢复到临时路径进行检查

### 风险 5：脚本路径硬编码难维护
**对策：**
- 所有路径放入 `sources.json`
- 脚本只读清单，不散落写死路径

## 7. 验收标准

本方案落地后，应满足以下标准：

1. 所有需备份的 `.claude` 目录都已登记到 `manifest/sources.json`
2. 集中仓库结构清晰反映四层架构
3. 同步脚本可一键执行，且可重复运行
4. Git 能显示每次同步的差异和历史
5. 快照归档按既定策略生成
6. 至少完成一次恢复演练
7. 能从集中仓库中恢复任意一个层级的 `.claude` 配置
