# Init Project

这个 skill 用于为项目智能初始化 `.claude` 目录，生成**简洁、可维护、符合项目实际情况**的配置，让后续 Claude 会话可以更快理解项目并执行正确工作流。

---

## 何时使用

在以下情况使用这个 skill：

- 用户直接说 `init-project`
- 用户说"智能初始化项目"
- 用户希望根据项目技术栈自动生成 `.claude` 配置
- 用户希望创建项目级规则、开发工作流规则、构建测试规则
- 用户希望快速建立项目专属的 Claude 工作流

在以下情况**不要**使用这个 skill：

- 用户只想创建最小 `.claude` 目录
- 用户只想手动建立基础目录结构
- 用户已经有完善 `.claude` 配置，只是要微调某个文件

---

## 核心目标

生成的配置应该**对当前项目真正有帮助**，而不是机械复制通用模板。

应该做到：

1. 分析目标仓库的类型、技术栈和目录结构
2. 创建合理的 `.claude` 目录结构
3. 优先生成项目最需要的规则文件
4. 保持配置精简，避免过度设计
5. 尊重并复用已有配置，而不是盲目覆盖

---

## 初始化前的分析

在写任何文件之前，先检查仓库，判断：

### 1. 项目类型

- 前端项目
- 后端项目
- 全栈项目
- 组件库 / SDK / 工具库
- 自动化脚本仓库
- monorepo

### 2. 技术栈

识别主要语言和框架：

- TypeScript / JavaScript
- React / Vue / Next.js / Nuxt
- Node.js / Python / Go / Rust

### 3. 工具链

识别：

- 包管理器（npm / pnpm / yarn / bun）
- 构建工具（vite / webpack / rollup）
- 测试框架（vitest / jest / pytest）
- lint / format 工具
- CI/CD 配置

### 4. 目录结构

识别：

- 核心源码目录
- 测试目录
- 配置文件目录
- 文档目录

### 5. 现有 `.claude` 状态

检查：

- 是否已经存在 `.claude`
- 是否已有规则文件
- 现有内容是否应保留、合并或仅补充

---

## 优先寻找的强信号

优先根据这些文件判断项目特征：

- `package.json`
- `pnpm-lock.yaml` / `yarn.lock` / `bun.lockb`
- `tsconfig.json`
- `vite.config.*` / `next.config.*` / `nuxt.config.*`
- `pyproject.toml` / `requirements.txt`
- `Cargo.toml` / `go.mod`
- `docker-compose.yml` / `Dockerfile`
- `README.md`
- `Makefile`
- `.github/workflows/`

如果这些信号不足以说明项目，应保持保守，只生成最小有用配置。

---

## 目录结构

项目层 `.claude` 通常包含：

```text
项目根目录/
├── CLAUDE.md              ← 项目入口，Claude 每次会话自动读取
└── .claude/
    ├── agents/            ← 项目专属 sub-agent 定义
    ├── rules/
    │   ├── common/
    │   └── specific/
    ├── plans/
    ├── temp/
    ├── settings/
    └── templates/
```

但不要为了"完整"而把所有目录都创建出来。

默认原则：

- `CLAUDE.md`（项目根目录）必须创建
- `.claude/` 下默认创建三个目录：`agents/`、`rules/`、`plans/`、`temp/`
- `settings/`、`templates/` 等目录视项目需要再创建
- 不要自动创建 `skills/` 目录
- 能少则少，后续可以再增量补充

---

## 优先创建的文件

### 0. 项目根目录 `CLAUDE.md`（最高优先级）

这是 Claude 每次进入项目时自动读取的入口文件，放在**项目根目录**（不是 `.claude/` 内部）。

应包含：

- 项目简介（一句话说明项目是什么）
- 主要目录结构（只列关键目录，不要穷举）
- 常用命令（build / test / lint / dev server）
- 技术栈概览
- 指向 `.claude/rules/architecture.md` 的引用
- 注意事项或约定（如有）

保持简洁，控制在 50~100 行以内。目标是让 Claude 在 30 秒内理解项目全貌。

### 1. `rules/architecture.md`

这是优先级最高的文件。

应说明：

- 项目作用域
- 主要目录结构
- 规则文件索引
- 快速参考
- 沟通偏好（默认中文优先、简洁实用）

### 2. 项目特定规则文件

根据项目情况，按需创建：

- 构建与测试规则
- 代码结构说明
- 技术栈约束说明
- 项目开发工作流说明
- 项目特定工具或 MCP 使用说明

不要创建无意义的"占位规则"。

## 执行流程

### 步骤 1：检查仓库

先读取目录结构与关键配置文件，确认项目类型、技术栈、工具链和目录边界。

### 步骤 2：总结判断结果

在动手前，先总结：

- 识别出的项目类型
- 主要技术栈
- 计划创建或更新的文件
- 为什么要创建这些文件

### 步骤 3：创建最小但有用的结构

优先从这些开始：

- 项目根目录 `CLAUDE.md`（必须创建）
- `.claude/agents/`、`.claude/rules/`、`.claude/plans/`、`.claude/temp/`（默认创建）
- 1~3 个真正必要的项目规则文件（放入 `rules/`）
- 只有在项目确实需要时才补 `settings/`、`templates/`、`temp/` 等目录
- 不要自动创建 `skills/` 目录

### 步骤 4：验证初始化结果

完成后检查：

- 是否在项目根目录创建了 `CLAUDE.md`
- 是否创建了 `.claude/agents/` 目录
- `CLAUDE.md` 是否能让 Claude 在 30 秒内理解项目全貌
- 是否明确了主要目录和入口
- 是否清楚说明了 build / test / lint 的执行方式
- 是否与项目真实情况相符
- 是否避免了无关和冗余配置

---

## 输出要求

使用这个 skill 时，最终应输出：

1. 对项目类型的简短判断
2. 创建或更新了哪些文件（包括根目录 `CLAUDE.md`）
3. 每个关键文件的用途
4. 为什么这些文件适合当前项目
5. 后续建议补充哪些内容

---

## 与开发工作流的衔接

生成配置时，应让后续 Claude 更容易遵循这些工作方式：

- 研究优先
- 计划先行
- 重要结论持久化到文件
- 新功能或复杂变更优先规划
- 测试、验证、审查有明确入口

如果项目本身已有明显工作流（例如 `package.json` scripts、CI 流程、测试目录），应把这些真实信息写进项目规则，而不是凭空编造流程。

## 常见问题

### `.claude` 已存在

不要直接覆盖。

先读现有文件，判断哪些应保留、哪些应补充、哪些应谨慎调整。

### 路径无效或父目录不存在

在写文件前先验证路径有效性。

### 权限不足

如果无法创建目录或文件，应明确报告失败原因，不要留下半成品结构。

### 项目过于模糊

如果仓库结构无法说明其用途或技术栈，只创建最小版本的 `architecture.md`，并说明后续需要哪些信息才能继续增强。

---

## 最佳实践

- 先读项目，再定结构
- 优先创建项目特定规则，而不是通用样板
- 优先少量高价值文件，而不是大而全
- 优先增量完善，而不是一把重写
- 优先让后续 Claude 能快速理解项目
- 保持中文优先、表达简洁、内容实用
