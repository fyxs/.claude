# 个人开发规范

## 适用范围

本规范适用于 `personal-projects` 下的所有个人项目开发。

---

## 核心原则

- 🚀 **快速迭代优先**：先实现功能，再优化
- 🧪 **鼓励实验**：可以尝试新技术和新方案
- 📝 **文档可选**：根据项目需要决定文档详细程度
- 🎨 **个人风格**：代码风格以个人习惯为主

---

## 代码质量

### 基本要求

- ✅ 代码可读性良好
- ✅ 关键逻辑有注释
- ✅ 避免明显的安全漏洞
- ⚠️ 代码审查可选（复杂功能建议使用）

### 推荐实践

- 使用 code-reviewer agent（可选）
- 复杂功能进行安全检查
- 重要项目保持测试覆盖

---

## 提交规范

### 灵活的 Commit Message

可以使用简化格式：

```
简短描述即可

示例：
- 添加用户登录
- 修复样式问题
- 重构数据处理逻辑
```

也可以使用标准格式（推荐）：
```
feat: 添加用户登录功能
fix: 修复样式问题
refactor: 重构数据处理逻辑
```

---

## 测试要求

### 灵活的测试策略

- ⚠️ 测试覆盖率无强制要求
- ✅ 核心功能建议编写测试
- ✅ 实验性功能可以跳过测试
- ✅ 重要项目建议 TDD

### 测试类型

根据项目重要性选择：
- 简单项目：手动测试即可
- 中等项目：单元测试 + 手动测试
- 重要项目：完整测试套件

---

## 文档要求

### 最小化文档

**必需**：
- README.md（基本说明）
  - 项目是什么
  - 如何运行
  - 基本使用

**可选**：
- API 文档
- 组件文档
- 详细的使用指南

---

## 项目结构

### 灵活的目录组织

可以根据项目特点自由组织，建议保持基本清晰度：

```
src/
├── components/    # 组件（如适用）
├── utils/         # 工具函数
├── api/           # API 接口（如适用）
└── ...            # 其他根据需要添加
```

---

## 技术选择

### 自由探索

- ✅ 可以尝试新框架和库
- ✅ 可以使用实验性技术
- ✅ 可以采用非主流方案
- ⚠️ 注意评估技术风险

### 推荐实践

- 记录技术选型理由
- 评估学习成本
- 考虑长期维护性

---

## 安全要求

### 基本安全

即使是个人项目，也要注意：
- ❌ 不要提交敏感信息（密钥、密码）
- ✅ 使用环境变量管理配置
- ✅ 基本的输入验证

---

## 版本管理

### 简化的分支策略

```
main          # 主分支
├── feature-xxx  # 功能分支（可选）
└── experiment-xxx  # 实验分支
```

或者直接在 main 分支开发（小项目）

---

## 发布流程

### 灵活的发布

- 版本号可选
- 发布前基本测试
- 可以快速迭代

---

## 学习和实验

### 鼓励的实践

1. **技术探索**
   - 尝试新的框架和库
   - 实验新的架构模式
   - 探索最佳实践

2. **快速原型**
   - 快速验证想法
   - 不必追求完美
   - 重点在于学习

3. **记录经验**
   - 记录遇到的问题
   - 总结学到的知识
   - 分享有价值的发现

---

## 何时提升标准

当个人项目变得重要时，考虑采用企业级标准：
- 项目要公开发布
- 项目有其他用户
- 项目需要长期维护
- 项目涉及敏感数据

此时可以参考 `ts-projects` 的企业开发规范。
