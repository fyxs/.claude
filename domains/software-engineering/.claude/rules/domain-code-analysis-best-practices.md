# 代码分析最佳实践

> 规则 ID: DOMAIN-STANDARD-001
> 版本: 1.0.0
> 创建日期: 2026-03-09
> 最后更新: 2026-03-09
> 状态: 发布

> 本规则定义了分析代码库和技术文档时的核心原则，避免信息理解偏差。

## 核心原则

### 1. 代码是真相（Code is Truth）

**原则**：以实际代码为准，而非文档或注释。

**理由**：
- 文档可能过时
- 注释可能不准确
- 代码是唯一的真实状态

**实践**：
```bash
# ❌ 错误做法
看到文档说"组件库尚未提供" → 直接相信 → 标记为阻塞

# ✅ 正确做法
看到文档说"组件库尚未提供" → 搜索代码验证 → 确认实际状态
```

---

### 2. 主动验证原则（Active Verification）

**原则**：看到关键信息（尤其是阻塞性信息）时，立即搜索代码验证。

**触发条件**：
- "尚未提供"、"不支持"、"缺失"等阻塞性描述
- "需要等待"、"依赖XX"等依赖性描述
- 任何可能影响项目进度的关键信息

**验证步骤**：
1. 搜索相关代码
2. 查看实际使用情况
3. 确认真实状态
4. 更新理解

---

### 3. 命名规则推理（Naming Pattern Inference）

**原则**：根据已知的命名规则，推测未知组件/API 的名称。

**示例**：

**已知信息**：
- `TeDepartmentV2`（部门选择 v2）
- `TeEmployeeV2`（人员选择 v2）

**推理规则**：
- 格式：`Te + 英文名 + V2`
- 推测：
  - 岗位 → `TeJobV2`
  - 标签 → `TeLabelV2`
  - 职务 → `TeTitleV2`

**验证方法**：
```bash
# 搜索推测的组件名
grep -r "TeJobV2\|TeLabelV2\|TeTitleV2" src
```

---

### 4. 全面搜索策略（Comprehensive Search）

**原则**：不要只搜索已知的，要搜索可能的。

**搜索层级**：

**Level 1：精确搜索**（已知名称）
```bash
grep -r "TeDepartmentV2" src
```

**Level 2：模式搜索**（命名规则）
```bash
grep -r "Te.*V2" src
```

**Level 3：广泛搜索**（相关概念）
```bash
grep -r "部门\|人员\|岗位" src
```

---

### 5. 已有代码优先（Existing Code First）

**原则**：分析已有代码的实现方式，作为新功能的参考。

**步骤**：
1. 搜索类似功能的实现
2. 分析实现模式
3. 提取可复用的方案
4. 应用到新功能

**示例**：
```bash
# 发现某个文件已经使用了 v2 组件
# → 查看该文件的实现方式
# → 作为其他文件改造的参考
```

---

### 6. 保持文件内一致性（File-Level Consistency）

**原则**：在修改文件前，先分析文件中已有的类似代码，保持同一文件内的一致性。

**理由**：
- 不同文件可能使用不同的策略或模式
- 同一文件内应该保持一致的风格和实现方式
- 机械套用其他文件的模式可能导致不一致

**实践步骤**：

1. **分析已有模式**
   ```bash
   # 在修改文件前，先查看文件中已有的类似组件
   # 例如：如果要添加 TeLabelV2，先看 TeDepartmentV2 是如何使用的
   ```

2. **识别策略差异**
   ```typescript
   // 文件 A：使用 campusIds（院区隔离）
   <te-department-v2
     organizationId=""
     :campusIds="selectRegions"
   />

   // 文件 B：使用 organizationId（组织隔离）
   <te-department-v2
     :organizationId="organizationId"
   />
   ```

3. **保持一致性**
   - 同一文件中的类似组件应该使用相同的策略
   - 不要在同一文件中混用不同的模式

**错误示例**：
```typescript
// ❌ 错误：同一文件中使用不同的策略
<te-department-v2 :organizationId="organizationId" />
<te-label-v2 organizationId="" :campusIds="campusIds" />
```

**正确示例**：
```typescript
// ✅ 正确：同一文件中保持一致
<te-department-v2 :organizationId="organizationId" />
<te-label-v2 :organizationId="organizationId" />
```

**检查清单**：
- [ ] 已查看文件中已有的类似组件/代码
- [ ] 已理解文件使用的策略和模式
- [ ] 新增/修改的代码与文件内已有代码保持一致
- [ ] 没有机械套用其他文件的模式

## 实践检查清单

**分析新项目/新需求时**：

- [ ] 阅读文档，但不完全相信
- [ ] 搜索代码，验证文档描述
- [ ] 根据命名规则推测可能的组件/API
- [ ] 查找已有的类似实现
- [ ] 以代码实际情况为准

**遇到阻塞性信息时**：

- [ ] 立即搜索代码验证
- [ ] 不要直接标记为阻塞
- [ ] 确认真实状态后再评估
- [ ] 更新风险评估

**分析组件/API 时**：

- [ ] 搜索已知的组件名称
- [ ] 根据命名规则推测其他组件
- [ ] 搜索模式匹配（如 `Te.*V2`）
- [ ] 查看组件库文档验证
- [ ] 查找已有代码的使用示例

## 常见陷阱

### 陷阱 1：过度信任文档

**问题**：
- 文档说"组件库尚未提供"
- 直接相信，标记为阻塞

**实际**：
- 组件已经存在
- 项目中已经在使用

**教训**：
- 文档可能过时
- 必须验证代码

---

### 陷阱 2：缺少主动验证

**问题**：
- 看到关键信息不验证
- 直接基于文档做判断

**实际**：
- 代码状态与文档不符
- 导致错误的风险评估

**教训**：
- 关键信息必须验证
- 代码是唯一真相

---

### 陷阱 3：搜索范围过窄

**问题**：
- 只搜索已知的组件名
- 没有根据规则推测

**实际**：
- 存在其他相关组件
- 错过了可用的资源

**教训**：
- 使用模式搜索
- 根据规则推测

---

## 总结

**核心思想**：
1. 代码是真相，文档可能过时
2. 主动验证，不要盲目相信
3. 推理规律，全面搜索
4. 参考已有，复用方案

**记住**：
- 文档是参考，代码是真相
- 验证优于假设
- 搜索优于猜测
- 复用优于重写
