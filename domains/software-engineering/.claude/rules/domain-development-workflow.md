## 开发实施工作流

### 0. 研究与复用 _(任何新实现前必须执行)_

**优先级顺序：**

1. **GitHub 代码搜索优先**
   - 运行 `gh search repos` 和 `gh search code`
   - 在编写任何新代码前查找现有实现、模板和模式
   - 寻找可以解决 80%+ 问题的开源项目

2. **Exa MCP 研究**
   - 在规划阶段使用 `exa-web-search` MCP
   - 用于更广泛的研究、数据摄取和发现先前技术
   - 查找最佳实践和行业标准

3. **检查包注册表**
   - 在编写工具代码前搜索 npm、PyPI、crates.io 等
   - 优先使用经过实战检验的库而非手写解决方案
   - 评估库的维护状态、社区支持和安全性

4. **寻找可适配的实现**
   - 寻找可以 fork、移植或包装的开源项目
   - 优先采用或移植经过验证的方法，而非编写全新代码
   - 在现有结构内迭代

**复用原则：**
- 如果现有解决方案满足需求，优先采用
- 避免重新发明轮子
- 站在巨人的肩膀上

**研究产物持久化（MANDATORY）：**

研究结果必须写入持久化文件，不能只在聊天中口头总结。

- 输出文件：项目目录下的 `research.md`（这里是示例，文件名按主题或功能等命名，如 `research-auth.md`）
- 研究指令要使用深度词汇："deeply"、"in great details"、"intricacies"，避免浮于表面
- 人审查 `research.md` 后才能进入计划阶段
- 如果研究有误，必须纠正后再继续

**为什么持久化？**

- 聊天中的研究结果会在上下文压缩时丢失
- 文件是人审查和纠正的载体
- 防止"垃圾进，垃圾出"——研究错误会传导到计划和实现

### 1. 优先规划 + Annotation Cycle

**规划步骤：**

1. **使用 planner agent** 创建实施计划
2. **生成 `plan.md` 文件**（编码前），包含：
   - 实现方案和代码片段
   - 需要修改的文件路径
   - 权衡分析和注意事项
   - 分阶段的任务列表
3. **识别依赖和风险**

**CRITICAL：使用 `plan.md` 文件而非聊天讨论**

计划必须写入项目目录下的 `plan.md`（这里是示例，文件名按主题或功能等命名，如 `plan-auth.md`）。
`plan.md` 是人和 Claude 之间的"共享可变状态"，比聊天式交互更结构化、更可追溯。

**Annotation Cycle（标注循环）：**

```
Claude 写 plan.md → 人在编辑器中审查 → 人加批注 → Claude 更新计划 → 满意？
                                                                    ├─ 否 → 继续标注
                                                                    └─ 是 → 生成任务列表 → 实现
```

1. Claude 生成 `plan.md` 后，人在编辑器中直接加批注
2. 批注可以是：纠正假设、拒绝方案、补充约束、提供领域知识
3. 人告知 Claude："我在文档中加了批注，处理所有批注并更新文档，**不要开始实现**"
4. Claude 读取并处理批注，更新 `plan.md`
5. 循环 1-6 次，直到人满意

**"don't implement yet" 防护（MANDATORY）：**

在计划未经人确认前，Claude 不得开始写代码。每轮标注循环都必须包含"不要开始实现"的显式指令。只有人明确说"开始实现"时，才进入实现阶段。

**批注示例：**
- "用 drizzle:generate 生成迁移，不要用原始 SQL" — 领域知识
- "不对，这应该是 PATCH 不是 PUT" — 纠正假设
- "删掉这一节，这里不需要缓存" — 拒绝方案
- "签名不能改，调用方适配，不是库适配" — 保护接口

**参考实现技巧：**

如果有好的开源参考实现，在请求计划时一并提供。Claude 有具体参考时效果远好于从零设计。

**任务列表：**

计划确认后、实现前，要求 Claude 在 `plan.md` 中添加详细的任务清单。实现过程中逐项标记完成，作为进度追踪。

### 1.5 标准化实现指令

**计划确认后，使用标准化 prompt 启动实现：**

```
全部实现。每完成一个任务或阶段，在 plan.md 中标记为已完成。
不要在所有任务完成前停下来。不要添加不必要的注释或 jsdoc。
不要使用 any 或 unknown 类型。持续运行类型检查，确保不引入新问题。
```

**这条指令编码了关键约束：**
- "全部实现" — 执行计划中的所有内容，不要挑选
- "标记为已完成" — plan.md 是进度的唯一真相来源
- "不要停下来" — 不要中途暂停等待确认
- "不要添加不必要的注释" — 保持代码干净
- "不要使用 any" — 维持严格类型
- "持续运行类型检查" — 尽早发现问题

**核心理念：实现阶段应该是"无聊的"。**

所有创造性决策在 Annotation Cycle 中完成。到实现阶段，每个决策都已确认，执行应该是机械的、可预测的。

**实现阶段的反馈风格：**

实现过程中，人的角色从架构师转为监督者，反馈变得极其简短：
- "你没实现 deduplicateByTitle 函数"
- "这个页面应该在 admin 应用里，不是主应用，移过去"
- "wider"、"still cropped"、"2px gap"（前端视觉调整）

**方向错误时的处理：**

不要试图修补错误方向，直接 git revert 然后缩小范围重来。缩小范围后重做几乎总是比增量修复更好。

### 2. TDD 方法

**MANDATORY 工作流：**

1. **RED** - 先写测试
   - 定义预期行为
   - 测试应该失败（因为功能尚未实现）

2. **GREEN** - 实现以通过测试
   - 编写最小实现
   - 使测试通过

3. **IMPROVE** - 重构
   - 优化代码质量
   - 保持测试通过

4. **VERIFY** - 验证覆盖率
   - 确保 80%+ 覆盖率
   - 检查边缘情况

**Agent 支持：**
- 使用 **tdd-guide** agent 强制执行测试优先方法
- 对新功能和 bug 修复主动使用

---

### 3. 代码审查

**审查流程：**

1. **使用 code-reviewer agent**（编写代码后立即使用）
2. **处理问题优先级：**
   - CRITICAL - 必须修复
   - HIGH - 必须修复
   - MEDIUM - 尽可能修复
   - LOW - 可选

3. **审查重点：**
   - 代码质量和可读性
   - 安全漏洞
   - 性能问题
   - 最佳实践遵循

---

### 4. 提交与推送

**提交前检查：**
- 所有测试通过
- 代码审查问题已解决
- 文档已更新
- 无安全漏洞

**提交规范：**
- 详细的提交消息
- 遵循 conventional commits 格式
- 参见下方 [Git 工作流](#git-工作流) 了解详细格式

## 测试要求

### 最低测试覆盖率：80%

**测试类型（全部必需）：**

1. **单元测试**
   - 测试单个函数、工具、组件
   - 隔离测试，使用 mock
   - 快速执行

2. **集成测试**
   - 测试 API 端点
   - 测试数据库操作
   - 测试模块间交互

3. **E2E 测试**
   - 测试关键用户流程
   - 根据语言选择框架（Playwright、Cypress、Selenium 等）
   - 模拟真实用户场景

---

### 测试驱动开发（TDD）

**MANDATORY 工作流：**

| 步骤 | 动作 | 预期结果 |
|------|------|---------|
| 1. RED | 先写测试 | 测试应该失败 |
| 2. GREEN | 编写最小实现 | 测试应该通过 |
| 3. IMPROVE | 重构代码 | 测试保持通过 |
| 4. VERIFY | 验证覆盖率 | 达到 80%+ |

**TDD 优势：**
- 更好的代码设计
- 更少的 bug
- 更高的信心
- 活文档

---

### 测试失败故障排除

**故障排除步骤：**

1. **使用 tdd-guide agent** 获取帮助
2. **检查测试隔离**
   - 确保测试不相互依赖
   - 验证测试顺序不影响结果
3. **验证 mock 正确**
   - 检查 mock 数据
   - 验证 mock 行为
4. **修复实现，而非测试**（除非测试本身有错）

**常见问题：**
- 异步问题
- 时间依赖
- 外部依赖未 mock
- 测试数据污染

## 验证工作流

### 何时使用验证循环

**触发条件：**
- ✅ 完成功能或重要代码变更后
- ✅ 创建 PR 前
- ✅ 重构后
- ✅ 合并代码前
- ✅ 发布前

**不需要验证：**
- ❌ 简单的文档修改
- ❌ 注释更新
- ❌ README 编辑

---

### 验证阶段

#### 阶段 1：构建验证

**目的**：确保项目可以成功构建

**执行**：
```bash
# JavaScript/TypeScript
npm run build 2>&1 | tail -20

# Python
python -m build 2>&1 | tail -20
```

**判断**：
- ✅ 构建成功 → 继续下一阶段
- ❌ 构建失败 → **立即停止**，修复后再继续

---

#### 阶段 2：类型检查

**目的**：验证类型安全

**执行**：
```bash
# TypeScript
npx tsc --noEmit 2>&1 | head -30

# Python
pyright . 2>&1 | head -30
```

**判断**：
- 报告所有类型错误
- 修复 CRITICAL 错误后继续
- MEDIUM/LOW 错误可以稍后修复

---

#### 阶段 3：代码检查

**目的**：确保代码质量和风格一致

**执行**：
```bash
# JavaScript/TypeScript
npm run lint 2>&1 | head -30

# Python
ruff check . 2>&1 | head -30
```

**判断**：
- 修复所有 ERROR 级别问题
- WARNING 可以稍后处理

---

#### 阶段 4：测试套件

**目的**：验证功能正确性和覆盖率

**执行**：
```bash
# 运行测试并生成覆盖率报告
npm run test -- --coverage 2>&1 | tail -50
```

**判断**：
- ✅ 所有测试通过 + 覆盖率 ≥ 80% → 继续
- ❌ 有测试失败 → 修复后再继续
- ⚠️ 覆盖率 < 80% → 添加测试

---

#### 阶段 5：安全扫描

**目的**：检查安全漏洞和敏感信息

**执行**：
```bash
# 检查硬编码密钥
grep -rn "sk-\|api_key\|password\|secret" --include="*.ts" --include="*.js" src/ 2>/dev/null | head -10

# 检查 console.log
grep -rn "console.log" --include="*.ts" --include="*.tsx" src/ 2>/dev/null | head -10
```

**判断**：
- ❌ 发现硬编码密钥 → **立即修复**
- ⚠️ 发现 console.log → 清理或使用日志库

---

#### 阶段 6：变更审查

**目的**：审查所有变更

**执行**：
```bash
git diff --stat
git diff HEAD~1 --name-only
```

**审查重点**：
- 是否有意外的变更
- 是否缺少错误处理
- 是否考虑了边缘情况

---

### 验证报告

**报告格式**：
```
验证报告
========

构建：    [通过/失败]
类型：    [通过/失败] (X 个错误)
代码检查：[通过/失败] (X 个警告)
测试：    [通过/失败] (X/Y 通过, Z% 覆盖率)
安全：    [通过/失败] (X 个问题)
变更：    [X 个文件]

总体：    [准备就绪/需要修复]

待修复问题：
1. ...
2. ...
```

---

### 持续验证

**长会话中的验证策略：**

1. **设置检查点** - 每完成一个功能后验证
2. **定期验证** - 每 15 分钟或重大变更后
3. **提交前验证** - 每次 git commit 前

**快速验证命令**：
```bash
# 快速检查（构建 + 测试）
npm run build && npm test
```

## Git 工作流

### 提交消息格式

```
<type>: <description>

<optional body>
```

**类型（Types）：**
- `feat` - 新功能
- `fix` - Bug 修复
- `refactor` - 重构
- `docs` - 文档
- `test` - 测试
- `chore` - 杂项
- `perf` - 性能优化
- `ci` - CI/CD

**示例：**
```
feat: add user authentication

Implement JWT-based authentication with refresh tokens.
Includes login, logout, and token refresh endpoints.
```

**注意：** Attribution 已通过 ~/.claude/settings.json 全局禁用。

---

### Pull Request 工作流

**创建 PR 时：**

1. **分析完整提交历史**
   - 不仅仅是最新提交
   - 使用 `git log` 查看所有相关提交

2. **查看所有变更**
   - 使用 `git diff [base-branch]...HEAD` 查看所有变更
   - 确保理解每个变更的影响

3. **起草全面的 PR 摘要**
   - 描述变更的目的
   - 列出主要变更
   - 说明测试策略

4. **包含测试计划和 TODOs**
   - 列出已完成的测试
   - 标记任何待办事项

5. **推送时使用 -u 标志**（如果是新分支）
   - `git push -u origin feature-branch`

**PR 描述模板：**
```markdown
## 变更摘要
[简要描述变更]

## 主要变更
- 变更 1
- 变更 2

## 测试
- [ ] 单元测试
- [ ] 集成测试
- [ ] E2E 测试

## 待办事项
- [ ] 待办项 1
```

## 工作流集成

**完整流程：**

```
研究与复用 → 输出 research.md
    ↓ 人审查 research.md
优先规划 → 输出 plan.md
    ↓ Annotation Cycle（1-6 轮）
标准化实现指令
    ↓
TDD 方法（RED → GREEN → IMPROVE）
    ↓
代码审查
    ↓
验证循环
    ↓
Git 提交
    ↓
创建 PR
```

**关键原则：**
- 永远不要跳过研究阶段
- 研究结果必须持久化到 research.md
- 计划必须写入 plan.md，通过 Annotation Cycle 迭代
- 计划未确认前不得开始实现（"don't implement yet" 防护）
- 实现阶段应该是机械的，创造性决策在计划阶段完成
- 测试驱动开发
- 代码审查是必需的
- 验证后再提交
- 提交消息要有意义

---

## 最佳实践总结

1. **研究优先** - 在编写代码前搜索现有解决方案，结果写入 `research.md`
2. **规划驱动** - 计划写入 `plan.md`，通过 Annotation Cycle 反复迭代直到满意
3. **计划与实现分离** - 创造性决策在计划阶段完成，实现阶段应该是机械的
4. **测试优先** - TDD 不是可选的，是必需的
5. **审查严格** - 使用 code-reviewer agent 捕获问题
6. **验证完整** - 运行完整的验证循环
7. **提交清晰** - 编写有意义的提交消息
8. **PR 全面** - 提供完整的上下文和测试计划

**记住：** 质量优于速度。遵循这个工作流可以确保代码质量、可维护性和团队协作。
